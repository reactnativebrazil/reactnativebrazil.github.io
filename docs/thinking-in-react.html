Pensando em React Edit on GitHub

por Pete Caça

react é, na minha opinião, o caminho do premier para construir grandes aplicativos da Web rápidos com JavaScript. Ele escalou muito bem para nós no Facebook e Instagram.

Uma das muitas grandes partes do react é como ele faz você pensar sobre aplicativos como você construir eles. Neste post, eu vou levá-lo através do processo de pensamento de construir uma tabela de dados pesquisável usando o produto react.

Comece com um mock

Imagine que nós já temos uma API JSON e um mock do nosso designer. Nosso designer aparentemente não é muito bom, porque a simulação parece com isso:

Nossa API JSON retorna alguns dados que se parece com isso:

[
  {category: "Sporting Goods", price: "$49.99", stocked: true, name: "Football"},
  {category: "Sporting Goods", price: "$9.99", stocked: true, name: "Baseball"},
  {category: "Sporting Goods", price: "$29.99", stocked: false, name: "Basketball"},
  {category: "Electronics", price: "$99.99", stocked: true, name: "iPod Touch"},
  {category: "Electronics", price: "$399.99", stocked: false, name: "iPhone 5"},
  {category: "Electronics", price: "$199.99", stocked: true, name: "Nexus 7"}
];

 

Passo 1: quebrar a interface do usuário em um componente hierarquia

A primeira coisa que você vai querer fazer é desenhar caixas em torno de cada componente (e subcomponente) na simulação e dar-lhes todos os nomes. Se você está trabalhando com um designer, eles podem já ter feito isso, então vá falar com eles! Seus nomes de camada do Photoshop pode acabar sendo os nomes de seus componentes React!

Mas como você sabe o que deve ser o seu próprio componente? Basta usar as mesmas técnicas para decidir se você deve criar uma nova função ou objeto. Uma dessas técnicas é o princípio da responsabilidade individual, ou seja, um componente deve, idealmente, faz uma coisa. Se ele acaba crescendo, deve ser decomposto em subcomponentes menores.

Desde que você está muitas vezes exibindo um modelo de dados JSON para um usuário, você vai descobrir que se o modelo foi construído corretamente, a interface do usuário (e, portanto, sua estrutura de componente) irá mapear bem. Isso porque os modelos de interface do usuário e de dados tendem a aderir à mesma arquitetura de informação, o que significa que o trabalho de separar a interface do usuário em componentes é muitas vezes trivial. Apenas dividi-lo em componentes que representam exatamente um pedaço de seu modelo de dados.

Você vai ver aqui que nós temos cinco componentes em nosso aplicativo simples. Eu tenho em itálico os dados cada componente representa.

  1. FilterableProductTable (laranja): contém a totalidade do exemplo
  2. SearchBar (azul): recebe toda a entrada do usuário
  3. ProductTable (verde): monitores e filtra a coleta de dados com base na entrada do usuário
  4. ProductCategoryRow (turquesa): exibe um título para cada categoria
  5. ProductRow (vermelho): exibe uma linha para cada produto

Se você olhar para ProductTable você vai ver que o cabeçalho da tabela (que contém o "Nome" e "preço" etiquetas) não é o seu próprio componente. Esta é uma questão de preferência, e não há um argumento a ser feito de qualquer maneira. Para este exemplo, eu deixei-o como parte de ProductTable porque é parte de tornar a coleta de dados que é ProductTable responsabilidade 's. No entanto, se este cabeçalho cresce para ser complexa (ou seja, se fôssemos para adicionar affordances para classificar), ele certamente faria sentido fazer esta a sua própria ProductTableHeader componente.

Agora que já identificou os componentes na nossa simulação, vamos organizá-los em uma hierarquia. Isso é fácil. Componentes que aparecem dentro de outro componente no simulado deve aparecer como uma criança na hierarquia:

  • FilterableProductTable
    • SearchBar
    • ProductTable
      • ProductCategoryRow
      • ProductRow

Passo 2: Criar uma versão estática em react

Agora que você tem a sua hierarquia de componentes, é hora de implementar o seu aplicativo. A maneira mais fácil é construir uma versão que leva o seu modelo de dados e torna a interface do usuário, mas não tem interatividade. É melhor separar esses processos porque construir uma versão estática requer um monte de digitação e sem pensar, e adicionar interatividade requer um monte de pensamento e não um monte de digitação. Vamos ver o porquê.

Para construir uma versão estática do seu aplicativo que processa o seu modelo de dados, você vai querer construir componentes que reutilizam outros componentes e transmitir dados usando adereços. Adereços são uma forma de passar dados de pai para filho. Se você estiver familiarizado com o conceito de estado, não use estado em tudo para construir esta versão estática. Estado é reservado apenas para a interatividade, ou seja, dados que mudam ao longo do tempo. Uma vez que esta é uma versão estática do aplicativo, você não precisa dele.

Você pode construir de cima para baixo ou de baixo para cima. Ou seja, você pode começar com a construção dos componentes mais acima na hierarquia (ou seja, começando com FilterableProductTable ou com os inferiores nele ProductRow Nos exemplos mais simples, normalmente é mais fácil ir de cima para baixo, e em projetos maiores, é mais fácil ir de baixo para cima e escrever testes como você construir.

No final desta etapa, você terá uma biblioteca de componentes reutilizáveis ​​que prestam o seu modelo de dados. Os componentes apenas terá render() métodos já que esta é uma versão estática do seu aplicativo. O componente no topo da hierarquia FilterableProductTable terá o seu modelo de dados como um adereço. Se você fizer uma alteração em seu modelo de dados subjacente e chamar ReactDOM.render() novamente, a interface do usuário será atualizada. É fácil ver como a interface do usuário é atualizado e onde fazer alterações uma vez que não há nada de complicado acontecendo. De react de uma maneira de fluxo de dados (também chamado de ligação one-way) mantém tudo modular e rápido.

Basta referir aos docs react se você precisar de ajuda para executar esta etapa.

Um breve interlúdio: adereços vs estado

Existem dois tipos de dados "modelo" em react: adereços e estado. É importante entender a distinção entre os dois; skim o oficial react docs, se você não tem certeza qual é a diferença.

Passo 3: Identificar a representação mínima (mas completa) da interface do usuário do estado

Para tornar a sua interface de usuário interativa, você precisa ser capaz de desencadear alterações ao seu modelo de dados subjacente. react torna isso fácil com o estado.

Para construir seu aplicativo corretamente, você primeiro precisa pensar no conjunto mínimo de estado mutável que as suas necessidades de aplicativos. A chave aqui é DRY: Do not Repeat Yourself. Descobrir a representação absoluta mínima das suas necessidades de aplicação estaduais e calcular tudo o que você precisa on-demand. Por exemplo, se você está construindo uma lista TODO, basta manter uma matriz dos itens TODO redor; não manter uma variável de Estado separado para a contagem. Em vez disso, quando você deseja processar a contagem TODO, basta levar o comprimento da matriz itens TODO.

Pense em todas as partes de dados em nossa aplicação exemplo. Nós temos:

  • A lista original de produtos
  • O texto busca que o usuário inseriu
  • O valor da caixa de seleção
  • A lista filtrada de produtos

Vamos passar por cada um e descobrir qual é o estado. Basta perguntar a três perguntas sobre cada pedaço de dados:

  1. É aprovada em de um pai via adereços? Se assim for, provavelmente não é o estado.
  2. Isso muda com o tempo? Se não for, provavelmente não é do Estado.
  3. Você pode calcular-lo com base em qualquer outro estado ou adereços em seu componente? Se assim for, não do Estado.

A lista inicial de produtos é passado como adereços, de modo que não é Estado. O texto de pesquisa e na caixa de seleção parecem ser estado, uma vez que mudar ao longo do tempo e não pode ser calculado a partir de nada. E, finalmente, a lista filtrada de produtos não é o estado, porque pode ser calculado através da combinação da lista original de produtos com o texto da pesquisa e do valor da caixa de seleção.

Então, finalmente, o nosso estado é:

  • O texto busca que o usuário inseriu
  • O valor da caixa de seleção

Passo 4: Identificar onde o seu estado deve viver

OK, então nós identificamos que o conjunto mínimo de estado app é. Em seguida, precisamos identificar mutação que o compõem, ou detém, neste estado.

Lembre-se: react é toda sobre one-way fluxo de dados para baixo na hierarquia do componente. Pode não ser imediatamente claro qual componente deve possuir o estado Isso é muitas vezes a parte mais difícil para os recém-chegados para entender, por isso siga estes passos para descobrir isso.:

Para cada parte do estado na sua aplicação:

  • Identificar cada componente que torna algo com base nesse estado.
  • Encontre um componente proprietário comum (um componente único acima de todos os componentes que precisam do Estado na hierarquia).
  • Ou o proprietário comum ou outro componente mais acima na hierarquia deve possuir o estado.
  • Se você não consegue encontrar um componente em que faz sentido para o próprio estado, criar um novo componente simplesmente para manter o estado e adicioná-lo em algum lugar na hierarquia acima do componente comum proprietário.

Vamos percorrer essa estratégia para a nossa aplicação:

  • ProductTable precisa filtrar a lista de produtos com base no estado e SearchBar precisa exibir o texto de pesquisa e estado selecionado.
  • O componente comum proprietário é FilterableProductTable
  • É conceptualmente faz sentido para o texto do filtro e um valor marcado para viver em FilterableProductTable

Legal, então nós decidimos que nossas vidas estaduais em FilterableProductTable Primeiro, adicione um getInitialState() método para FilterableProductTable que retorna {filterText: '', inStockOnly: false} para refletir o estado inicial de sua aplicação. Em seguida, passe filterText e inStockOnly para ProductTable e SearchBar como um adereço. Finalmente, usar esses adereços para filtrar as linhas em ProductTable e defina os valores dos campos do formulário em SearchBar

Você pode começar a ver como o aplicativo irá se comportar: set filterText a "ball" e atualizar o seu aplicativo. Você verá que a tabela de dados é atualizado corretamente.

Passo 5: Adicionar inversa fluxo de dados

Até agora, nós construímos um aplicativo que processa corretamente como uma função de adereços e estado fluindo para baixo na hierarquia. Agora é hora de apoiar dados que fluem para o outro lado: os componentes do formulário profundas na hierarquia precisa atualizar o estado em FilterableProductTable

react torna este fluxo de dados explícita para tornar mais fácil para entender como o seu programa funciona, mas exige um pouco mais de digitação de ligação de dados tradicionais de duas vias. react fornece um add-on chamado ReactLink para fazer esse padrão tão conveniente como a ligação de duas vias, mas para o objetivo deste post, vamos manter tudo explícito.

Se você tentar digitar ou marque a caixa na versão atual do exemplo, você verá que react ignora sua entrada. Isso é intencional, como temos definir o value prop da input a ser sempre igual ao state passado do FilterableProductTable

Vamos pensar sobre o que queremos que aconteça. Nós queremos ter certeza de que sempre que o usuário muda a forma, nós atualizamos o estado para refletir a entrada do usuário. Dado que os componentes devem apenas atualizar seu próprio estado, FilterableProductTable vai passar uma chamada de retorno para SearchBar que dispara sempre que o estado deve ser atualizado. Podemos usar o onChange evento nas entradas para ser notificado da mesma. E a chamada de retorno aprovada pelo FilterableProductTable chamará setState() eo aplicativo será atualizado.

Embora isso soa complexo, é realmente apenas algumas linhas de código. E é realmente explícita como seus dados está fluindo ao longo do app.

E é isso

Esperemos que este lhe dá uma idéia de como pensar sobre a construção de componentes e aplicativos com react. Embora possa ser um pouco mais digitação do que você está acostumado, lembre-se de que o código é lido muito mais do que está escrito, e é extremamente fácil de ler isto, código explícito modular. Como você começar a construir grandes bibliotecas de componentes, você vai apreciar esta explicitação e modularidade, e com a reutilização de código, suas linhas de código vai começar a encolher. :)
